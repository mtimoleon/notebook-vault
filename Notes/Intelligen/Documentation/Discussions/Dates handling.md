Σε σχέση με τα dates: Κάπως έτσι τα σκεφτόμουν κι εγώ ανεξάρτητα από τα διάφορα άρθρα. Συνοψίζω τις 2 λύσεις:

- Χρησιμοποιούμε παντού DateTimeOffset: Τα dates δημιουργούνται πάντα στο TZ του server και έτσι αποθηκεύονται και στη βάση.
- Χρησιμοποιούμε DateTime: Στη βάση πρέπει να σώζουμε χωριστά το TZ, ενώ για λόγους ευκολίας θα μπορούσαμε να τα σώζουμε όλα σε UTC (άρα θα πρέπει κατά το save να μεταφράζουμε σε UTC και συνεπώς κατά την ανάγνωση να μεταφράζουμε πάλι σε local time).

Τίθεται βέβαια κι ένα άλλο θέμα: είναι πολύ πιθανό στην ίδια εφαρμογή να φτάχνονται schedules για διαφορετικά εργοστάσια που μπορεί να είναι σε διαφορετικά TZs, άρα το να έχουν όλα τα dates το TZ του server δεν θα μας εξυπηρετούσε. Για τη δημιουργία των schedules έχω σκεφτεί τη λύση των workspaces, οπότε μια λύση θα ήταν κάθε workspace να καθορίζει το TZ του, ώστε κατά την ανάγνωση να μπορεί κάποιος να επιλέξει π.χ. μεταξύ: Local Time του user που διαβάζει τα δεδομένα, Local Time του workspace (δηλαδή το TZ του εργοστασίου), και UTC.Αν μπούμε στη διαδικασία να κάνουμε μόνοι μας handling του TZ μέσω των worskpaces, τότε δεν βλέπω σε αυτή τη φάση μεγάλη χρησιμότητα να πάμε σε DateTimeOffset, γιατί ακόμη και σε αυτή την περίπτωση θα έπρεπε να φροντίζουμε το TZ των νέων dates να συμφωνεί με αυτό του workspace.Αν λοιπόν παμε σε απλό UTC στη βάση, πρέπει να βρούμε κάποιον universal αυτοματοποιημένο τρόπο να μετατρέπουμε τα dates σε UTC πριν το saving γιατί αλλιώς θα έχουμε πολλά λάθη.Food for thought!
